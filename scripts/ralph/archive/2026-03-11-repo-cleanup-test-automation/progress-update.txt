## Codebase Patterns

- migrate.js usa pg diretamente (não prisma CLI) — evita dependência do effect/fast-check
- Prisma 7 CLI precisa de 'effect' + 'fast-check' (cadeia pesada, não incluir no Docker runner)
- Docker standalone build copia apenas o que está explicitamente no COPY do Dockerfile
- O postgres usa bind mount em /opt/quayer/data/postgres — NÃO deletar no git clean
- git clean -fd -e .env -e data/ -e backups/ é o padrão correto no deploy script
- migrate.js verifica _prisma_migrations.finished_at IS NOT NULL para saber quais migrations já foram aplicadas
- Para pré-marcar migrations como aplicadas: INSERT INTO _prisma_migrations com finished_at definido
- gen_random_uuid() está disponível no Postgres 13+ (estamos no 15)
- **IMPORTANTE:** prisma migrate diff não funciona no Windows MINGW64 - usar geração manual de SQL ou executar em Linux/Docker
- Git em OneDrive no Windows pode ter erros de mmap - considerar mover repositório para fora do OneDrive
- ci.yml tem permissão corrompida no filesystem Windows — usar `git show HEAD:.github/workflows/ci.yml` para ler
- vitest.config.ts e vitest.config.integration.ts eram idênticos — manter apenas vitest.config.ts
- docker-compose.quayer.yml é o único compose válido — docker-compose.yml e docker-compose.prod.yml são duplicados
- test/ será deletado completamente — nova estrutura: test/unit/, test/api/, test/e2e/
